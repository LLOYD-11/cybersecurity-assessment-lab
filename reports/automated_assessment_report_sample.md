### F-001: SQL Injection in Vulnerable Login Flow

**Severity:** High

**Category:** Web Application Security

**Evidence:**

The vulnerable login route formats user input directly into a SQL query, allowing the payload `' OR '1'='1' --` to alter the query logic and bypass the password condition.

**Impact:**

In a real application, this pattern could allow authentication bypass, unauthorized data access, data modification, or data deletion.

**Remediation:**

Use parameterized queries so user input is treated as data rather than SQL syntax.

**Verification:**

The secure login route uses parameterized queries and rejects the same SQL injection payload.

**References:**

- OWASP Top 10 A03:2021 - Injection

### F-003: Plaintext Password Storage

**Severity:** High

**Category:** Authentication Security

**Evidence:**

The weak authentication flow stores and displays the raw password value for the demo account.

**Impact:**

If a database storing plaintext passwords is exposed, every stored password is immediately compromised. Password reuse can increase the impact beyond the original application.

**Remediation:**

Store salted password hashes using a password hashing function designed for authentication. Never store plaintext passwords.

**Verification:**

The secure authentication flow stores a salted Werkzeug password hash and verifies login attempts with `check_password_hash`.

**References:**

- OWASP Top 10 A07:2021 - Identification and Authentication Failures

### F-002: Cross-Site Scripting in Vulnerable Comment Flow

**Severity:** Medium

**Category:** Web Application Security

**Evidence:**

The vulnerable comment route renders untrusted input with Jinja's `safe` filter, allowing a submitted `<script>` payload to be interpreted by the browser.

**Impact:**

In a real application, cross-site scripting could allow JavaScript execution in another user's browser, session theft, page modification, or actions performed as the victim.

**Remediation:**

Keep template autoescaping enabled and do not mark untrusted user input as safe. If rich text is required, sanitize it with a strict allowlist.

**Verification:**

The secure comment route uses default escaping and renders the script payload as text.

**References:**

- OWASP Top 10 A03:2021 - Injection
